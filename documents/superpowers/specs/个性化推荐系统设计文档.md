# 博客个性化推荐系统 — 设计文档

## 1. 项目背景与目标

### 1.1 现状分析

博客系统已有的推荐相关能力：

- **热门文章排序**：`PopularArticleService` 基于加权评分（viewCount × 1 + likeCount × 10 + favoriteCount × 20），Redis 缓存 + DB 降级
- **相关文章推荐**：基于共同标签数量排序、同分类文章推荐（仅在文章详情页）
- **用户行为数据**：`UserArticle` 表记录点赞/收藏（布尔标记），文章表有 viewCount/likeCount/favoriteCount 聚合字段
- **基础设施**：Spring Boot 2.7 + MyBatis + Redis + MySQL，已有定时任务和异步处理模式

### 1.2 目标

在首页增加"为你推荐"模块，实现：

- **匿名用户**：基于热度和内容的推荐
- **登录用户**：基于个人行为的个性化推荐
- **性能要求**：请求响应 < 10ms（缓存命中），不引入额外中间件

## 2. 技术选型过程

### 2.1 用户模式选择

| 方案 | 描述 | 结论 |
|------|------|------|
| 仅匿名 | 推荐基于内容相似度和热度 | ❌ 无法个性化 |
| 仅登录 | 基于个人行为做推荐 | ❌ 覆盖面窄 |
| **混合模式** | **匿名看热度，登录看个性化** | ✅ 采用 |

**决策理由**：博客同时有匿名访客和注册用户，混合模式能覆盖所有场景，且匿名推荐可以作为登录用户的降级方案

> 备注：匿名用户



### 2.2 复杂度选择

| 方案 | 技术栈 | 额外依赖 | 结论 |
|------|--------|----------|------|
| 轻量级 | 规则推荐（标签匹配+热度） | 无 | ❌ 效果有限 |
| **中等** | 行为追踪 + 协同过滤 + 内容推荐 | 无（Java+Redis+MySQL） | ✅ 采用 |
| 重量级 | Elasticsearch / ML 模型 | ES/Python | ❌ 过度设计 |

**决策理由**：个人博客场景下，**数据量有限**（文章 < 1万，用户 < 1万），中等方案<u>在不引入额外基础设施的前提下</u>能<u>提供足够好的推荐效果</u>。协同过滤在小规模数据上表现良好，TF-IDF 分词需要引入 HanLP/jieba 等中文分词库，性价比不高。

### 2.3 推荐算法选型

对比了三种方案：

**方案 A：标签内容推荐 + 简单协同过滤**
- 怎么做？
  - 内容相似度：直接用文章的**标签集合**计算 Jaccard 相似度（即两个标签集合的交集 / 并集）。
  - 协同过滤：只用用户行为中的**点赞/收藏**数据（信号强、噪声少），做简单的基于物品的协同过滤（Item-CF）。
- **优点**：极简！代码量少，无需 NLP 分词，开发快，运维成本低。
- **缺点**：**信号稀疏**——如果用户很少点赞/收藏，或者文章标签不全/不规范，协同过滤就失效了；推荐结果可能单调（只推热门或标签匹配的文章）

> 💡 案例：用户 A 点赞了《Spring Boot 入门》，系统只看他点赞过的文章标签，找其他也有 “Spring Boot” 标签的文章推荐给他。但如果他没点过赞，系统就完全“看不见”他的兴趣。

> 什么是“用文章的**标签集合**计算 Jaccard 相似度”：**拿两篇文章的标签列表，算它们“重合比例”。比例越高，越相似**
>
> 类比——两个朋友的爱好列表：
>
> - 小明：打篮球、看电影、吃火锅
> - 小红：打篮球、唱歌、吃火锅
>   → 共同爱好：打篮球、吃火锅（2个）
>   → 总爱好：打篮球、看电影、吃火锅、唱歌（4个）
>   → 相似度 = 2/4 = 50%

**方案 B：TF-IDF 内容相似度 + 行为加权协同过滤**

- 怎么做？
  - 内容相似度：对文章正文做 **TF-IDF 向量化**（需要分词 → 构建词频矩阵 → 计算权重），再算余弦相似度。
  - 协同过滤：用**用户行为评分矩阵**（如：浏览=1分，点赞=3分，收藏=5分），做更精细的加权协同过滤。
- **优点**：内容理解更深——即使标签缺失，也能从正文挖掘语义；行为加权让推荐更精准。
- **缺点**：**需要中文分词库**（如 jieba），增加了依赖和工程复杂度；分词质量直接影响效果；对小博客来说“杀鸡用牛刀”。

> 💡 案例：用户看了《如何用 Redis 缓存热点数据》，虽然没打标签，但 TF-IDF 能识别出关键词 “Redis”、“缓存”、“热点”，从而推荐《Redis 高并发实战》这类文章。

> ##### **🍳 举个栗子（超简单版）：**
>
> 假设你有3篇文章：
>
> | 文章 | 内容关键词（简化后）   |
> | :--- | :--------------------- |
> | A    | Java Java Spring Redis |
> | B    | Python Python AI ML    |
> | C    | Java Redis ML          |
>
> ##### **第一步：统计每个词在每篇文章里出现几次（*****\*TF = Term Frequency，词频\******）**
>
> - 文章A：Java=2, Spring=1, Redis=1
> - 文章B：Python=2, AI=1, ML=1
> - 文章C：Java=1, Redis=1, ML=1
>
> ##### **第二步：给常见词“打折”（*****\*IDF = Inverse Document Frequency，逆文档频率\******）**
>
> - “Java” 出现在 A 和 C → 2篇文章 → 不算太稀有 → 权重低一点
> - “AI” 只出现在 B → 很稀有 → 权重高！
>   → 这样，“AI” 就比 “Java” 更能代表文章B的独特性。
>
> ##### **第三步：把每篇文章变成一个“数字向量”**
>
> 比如文章A → `[Java: 0.8, Spring: 1.5, Redis: 1.2]`（具体数字是 TF×IDF 计算出来的）
> 文章C → `[Java: 0.4, Redis: 0.6, ML: 1.0]`
>
> ##### **第四步：算相似度（用*****\*余弦相似度\******）**
>
> - 就像比较两个箭头的方向：方向越接近，相似度越高。
> - 文章A 和 C 都有 Java 和 Redis → 箭头方向接近 → 相似度高
> - 文章A 和 B 几乎没共同词 → 箭头垂直 → 相似度≈0

> #### 加权协同过滤
>
> 加权 = 给不同行为打分
>
> | 行为 | 分值 | 含义         |
> | :--- | :--- | :----------- |
> | 浏览 | 1分  | “我看了”     |
> | 点赞 | 3分  | “我喜欢”     |
> | 收藏 | 5分  | “这是宝藏！” |
>
> **协同过滤怎么用这个分？**
>
> 想象一个表格（叫“用户-物品评分矩阵”）：
>
> | 用户 \ 文章 | 《Java并发》 | 《Spring Boot》 | 《Python入门》 |
> | :---------- | :----------- | :-------------- | :------------- |
> | 用户A       | 5（收藏）    | 3（点赞）       | 1（浏览）      |
> | 用户B       | 3（点赞）    | 5（收藏）       | 0（没看过）    |
> | 用户C       | 1（浏览）    | 0               | 5（收藏）      |
>
> 现在，你要给**新用户D**推荐文章：
>
> - 发现用户D 浏览了《Java并发》（1分）
> - 看表格：**喜欢《Java并发》的人**（用户A、B）也都喜欢《Spring Boot》
> - 而且用户A给《Spring Boot》打了3分，用户B打了5分 → 平均4分，很高！
> - 所以推荐《Spring Boot》给D
>
> > ✅ 这就是“加权协同过滤”：**不是只看“有没有行为”，而是看“行为有多重”**，让推荐更精准。
>
> 对比方案A的“简单协同过滤”（只认点赞/收藏=1，其他=0），就像：
>
> - 方案A：用户A点赞=1，用户B收藏=1 → 一样重视
> - 方案C：用户A点赞=3，用户B收藏=5 → B的行为更重要！



**方案 C（采用）：标签内容推荐 + 行为加权协同过滤**

| 组件       | 来源                                  | 权重 | 作用                                                     |
| ---------- | ------------------------------------- | ---- | -------------------------------------------------------- |
| 内容相似度 | 标签 Jaccard                          | 40%  | 保证基础相关性，零额外依赖（不用分词）                   |
| 协同过滤   | 行为加权评分矩阵（浏览1/点赞3/收藏5） | 40%  | 弥补方案 A 的“信号稀疏”问题，利用行为数据提升个性化      |
| 热度排序   | 文章阅读量/点赞数等全局指标           | 20%  | 保证推荐结果有“新鲜感”和“普适性”，避免冷启动时推荐太冷门 |

- 内容相似度用标签 Jaccard（零额外依赖）
- 协同过滤用行为加权评分矩阵（浏览1/点赞3/收藏5）
- 三路混合：内容 40% + 协同过滤 40% + 热度 20%

> **最终决策理由**：方案 C 取了 A 和 B 的优点 — 不引入分词依赖（避免方案 B 的复杂度），但用行为加权评分替代简单布尔标记（弥补方案 A 信号稀疏的问题）。三路混合保证了推荐的多样性和鲁棒性。
>
> **为什么内容相似度使用“标签 Jaccard”？占40%权重？**
>
> 标签 Jaccard：用文章的标签列表，算两篇文章的“重合比例。
>
> 在方案C中，它负责 **40% 的推荐权重**，原因：
>
> - 实现超级简单（你已有标签系统，直接用）
> - **不依赖用户行为（新文章、新用户也能推荐）**
> - 是整个系统的“压舱石”，保证基础相关性
>
> 而另外 40% + 20% 用行为加权 + 热度来补充，让它不单调、不冷启动失败





### 2.4 更新策略选择

| 方案 | 描述 | 结论 |
|------|------|------|
| 实时计算 | 每次请求实时算 | ❌ 性能开销大 |
| 定时预计算 | 全部定时批量算 | ❌ 个性化响应慢 |
| **混合策略** | 匿名用户 定时算，登录用户 行为触发异步更新 | ✅ 采用 |

**决策理由**：匿名用户的热度推荐变化慢，定时预计算（每小时）足够。登录用户需要行为响应性（刚收藏了一篇文章，下次刷新就该看到相关推荐），所以用事件驱动异步更新。请求时统一读 Redis 缓存，保证一致的低延迟。

## 4. 推荐算法详解

### 4.1 内容推荐引擎（Content-Based）— 权重 40%

**核心思想**：如果用户对标签 A 感兴趣，那么包含标签 A 的文章更可能被用户喜欢。

**文章相似度计算 — Jaccard 系数**：

```
Jaccard(文章X, 文章Y) = |Tags(X) ∩ Tags(Y)| / |Tags(X) ∪ Tags(Y)|
```

- 值域 [0, 1]，共同标签越多越相似
- 同分类额外加分 +0.2（上限 1.0）
- 示例：文章 X 有标签 {Java, Redis, Spring}，文章 Y 有标签 {Java, Spring, MySQL}
  - 交集 = {Java, Spring}，并集 = {Java, Redis, Spring, MySQL}
  - Jaccard = 2/4 = 0.5，若同分类则 0.5 + 0.2 = 0.7

**用户兴趣匹配**：

```
content_score(用户U, 文章A) = Σ(tag ∈ Tags(A)) user_interest_weight(U, tag) / |Tags(A)|
```

即用户对文章各标签的兴趣权重平均值。

### 4.2 协同过滤引擎（Item-Based CF）— 权重 40%

**核心思想**：如果很多用户同时喜欢文章 A 和文章 B，那么喜欢 A 的用户也可能喜欢 B。

**用户-文章评分矩阵**：

| | 文章1 | 文章2 | 文章3 | 文章4 |
|---|---|---|---|---|
| 用户A | 1(浏览) | 5(收藏) | - | 3(点赞) |
| 用户B | 3(点赞) | - | 1(浏览) | 5(收藏) |
| 用户C | - | 5(收藏) | 3(点赞) | - |

评分规则：浏览 = 1（阅读 > 30s 则 = 2），点赞 = 3，收藏 = 5。同一用户对同一文章取最高分。

**物品相似度 — 余弦相似度**：

```
sim(A, B) = Σ(u ∈ Users(A)∩Users(B)) rating(u,A) × rating(u,B)
            / (√Σ rating(u,A)² × √Σ rating(u,B)²)
```

仅计算对两篇文章都有行为的用户子集。当文章行为数据不足 5 条时不参与协同过滤（数据太稀疏，结果不可靠）。

**为用户生成推荐**：

```
cf_score(用户U, 文章A) = Σ(B ∈ 用户U已交互文章) rating(U,B) × sim(A,B)
                        / Σ(B) |sim(A,B)|
```

即用户已交互文章与候选文章的加权相似度。

### 4.3 热度引擎（Popularity）— 权重 20%

复用现有 `PopularArticleService` 评分公式，增加时间衰减：

```
raw_score = viewCount × 1 + likeCount × 10 + favoriteCount × 20
popularity_score = raw_score × e^(-0.01 × 发布天数)
```

时间衰减因子让新文章有更多曝光机会，避免老文章因累积数据永远霸榜。

### 4.4 三路混合与归一化

```
final_score = 0.4 × normalize(content_score)
            + 0.4 × normalize(cf_score)
            + 0.2 × normalize(popularity_score)
```

- 各路得分先 min-max 归一化到 [0, 1]
- 同一篇文章在多路出现时取最高分
- 过滤掉用户已浏览过的文章
- 取 Top 10 返回

### 4.5 冷启动策略

```
行为数量 ≥ 5  → 标准三路混合（0.4 + 0.4 + 0.2）
有兴趣标签    → 内容引擎 60% + 热度 40%（跳过协同过滤）
无任何数据    → 热度引擎 100%（等同匿名用户）
```

用户注册时可选择感兴趣的标签/分类，写入 `user_interest` 表（初始权重 0.5）。随着行为积累，兴趣权重动态调整，推荐逐步从内容驱动过渡到协同过滤驱动。

## 5. 数据模型设计

### 5.1 新增表

**user_behavior（用户行为记录表）**

```sql
CREATE TABLE user_behavior (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       BIGINT       NOT NULL,
    article_id    BIGINT       NOT NULL,
    behavior_type TINYINT      NOT NULL COMMENT '1=浏览 2=点赞 3=收藏',
    duration      INT          DEFAULT NULL COMMENT '阅读时长(秒)，仅浏览行为',
    create_time   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_behavior (user_id, behavior_type),
    INDEX idx_article_behavior (article_id, behavior_type),
    INDEX idx_create_time (create_time)
);
```

**user_interest（用户兴趣标签表）**

```sql
CREATE TABLE user_interest (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     BIGINT   NOT NULL,
    tag_id      BIGINT   NOT NULL,
    weight      DOUBLE   NOT NULL DEFAULT 0.5 COMMENT '兴趣权重 0-1',
    source      TINYINT  NOT NULL COMMENT '1=注册选择 2=行为推断',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE INDEX uk_user_tag (user_id, tag_id),
    INDEX idx_user (user_id)
);
```

### 5.2 Redis 缓存结构

| Key | 类型 | 内容 | TTL |
|-----|------|------|-----|
| `rec:user:{userId}` | ZSet | 个性化推荐（articleId → score） | 2h |
| `rec:anonymous` | ZSet | 匿名热度推荐 | 1h |
| `rec:cold:{userId}` | ZSet | 冷启动推荐 | 2h |
| `sim:content:{articleId}` | ZSet | 内容相似度 | 24h |
| `sim:cf:{articleId}` | ZSet | 协同过滤相似度 | 12h |
| `behavior:score:{userId}` | Hash | 用户评分（articleId → score） | 7d |

### 8.3 匿名用户推荐与热门文章差异化

推荐横条和侧边栏热门文章使用**不同策略**，避免内容重复：

| 维度 | 侧边栏热门文章 | 匿名用户推荐（首页横条） |
|------|---------------|------------------------|
| **算法** | 纯热度：`viewCount×1 + likeCount×10 + favoriteCount×20` | 三因子：`热度50% + 新鲜度30% + 多样性20%` |
| **时间衰减** | ❌ 无衰减，老文章可能永远霸榜 | ✅ 有衰减，新文章有机会曝光 |
| **多样性** | ❌ 仅看互动指标 | ✅ 看标签多样性，避免标签单一 |
| **展示数量** | Top 5 | Top 10 |
| **去重** | 无 | 排除侧边栏 Top 5，保证零重复 |
| **用途** | 首页侧边栏 | 首页推荐横条 |

**设计理由**：
- 侧边栏是"经典热门"——谁互动多谁上，适合快速浏览
- 推荐横条是<u>"新鲜 + 多样化"</u>的综合推荐——**热门的同时保证内容不单调，且不和侧边栏重复**

**算法详解**：

侧边栏热门文章（`PopularArticleServiceImpl.scoreAndTopN`）：
```
heatScore = viewCount × 1 + likeCount × 10 + favoriteCount × 20
```

匿名用户推荐（`RecommendServiceImpl.recomputeAnonymousRecommendations`）：
```
anonScore = popularityScore × 0.5        // 热度分
           + popularityScore × 0.3        // 新鲜度（已内置于热度分的时间衰减）
           + diversityScore × 0.2         // 标签多样性

其中 diversityScore = min(1.0, tagCount / 5)  // 标签越多多样性越高
```

> #### 多样性的直觉
>
> **标签数量 = 文章主题的广度**
>
> | 文章 | 标签                                 | 语义                     |
> | ---- | ------------------------------------ | ------------------------ |
> | A    | `Java`                               | 只讲 Java，很专一        |
> | B    | `Java, Spring, Redis, MySQL, Docker` | 涉及多个技术方向，更"杂" |
>
> 多样性分数的目的是：**不要让推荐结果全是同一类文章**。
>
> #### 为什么 `tagCount / 5`
>
> ```java
> diversityScore = min(1.0, tagCount / 5)
> ```
>
> - `tagCount = 1` → `diversityScore = 0.2` → 多样性很低
> - `tagCount = 3` → `diversityScore = 0.6` → 中等
> - `tagCount = 5` → `diversityScore = 1.0` → 达到上限，不再增加
> - `tagCount = 10` → `diversityScore = 1.0`（被 cap 住）
>
> **为什么是 5**：这是一个人为设定的阈值，假设一篇文章有 5 个标签就算"足够多样"了。超过 5 个也不再加分，避免少数超长标签列表的 article 权重过大。
>
> #### 在推荐中的作用
>
> ```java
> anonScore = popularityScore × 0.5  // 热度
>            + popularityScore × 0.3  // 新鲜度（内置于热度分的时间衰减）
>            + diversityScore × 0.2    // 标签多样性
> ```
>
> 多样性占 20% 权重。这意味着：
>
> - 两篇文章热度相同，标签多的那篇会排在前面
> - 推荐列表不会全是"Java 系列"，也会有"Java + Python + DevOps"的杂食文章
>
> **本质上是给"跨领域"文章一点额外曝光机会，避免推荐结果太垂直。**
>
> 

**去重逻辑**：

```java
// 推荐横条排除侧边栏 Top 5
List<ArticleShortVO> popularList = popularArticleService.getCachedOrFallback(5);
Set<Long> excludeIds = popularList.stream()
        .map(ArticleShortVO::getId)
        .collect(Collectors.toSet());
```

## 10. 面试高频问题预备

### Q1: 为什么选择 Item-Based CF 而不是 User-Based CF？

Item-Based CF 在博客场景下更合适：
- 文章数量远少于用户数量，item-item 相似度矩阵更小，计算和存储成本低
- 文章的属性相对稳定（不会频繁变化），相似度矩阵更新频率低
- User-Based CF 需要找相似用户，在用户行为稀疏时效果差
- Item-Based CF 的推荐结果可解释性更好（"因为你喜欢了文章A，推荐相似的文章B"）

### Q2: 如何解决冷启动问题？

三层降级策略：
1. 注册时引导用户选择兴趣标签 → 基于标签做内容推荐
2. 用户跳过兴趣选择 → 降级到热门推荐（等同匿名用户）
3. 随着行为积累（≥ 5 条），逐步切换到完整的三路混合推荐

### Q3: 如何保证推荐的实时性和性能？

混合更新策略：
- 匿名推荐：定时预计算（每小时），变化慢无需实时
- 个性化推荐：行为事件驱动异步更新，用户行为后几秒内缓存刷新
- 请求链路统一读 Redis，命中时 < 5ms
- 缓存未命中有同步计算兜底（500ms 超时），超时降级到热度推荐

### Q4: 三路混合的权重怎么确定的？为什么是 0.4/0.4/0.2？

- 内容推荐和协同过滤各 40%：两者互补 — 内容推荐解决协同过滤的冷启动问题，协同过滤能发现标签无法捕捉的隐含关联
- 热度 20%：保证推荐列表中有一定比例的高质量内容，避免推荐过于小众
- 权重可配置，后续可通过 A/B 测试调优

### Q5: 数据稀疏怎么办？

多重应对：
- 行为加权评分（浏览1/点赞3/收藏5）比布尔标记提供更丰富的信号
- 协同过滤设置最低行为阈值（5条），数据不足的文章不参与 CF
- 三路混合本身就是对稀疏性的对冲 — CF 数据不足时，内容推荐和热度推荐兜底
- 时间衰减让新文章有曝光机会，不会因为数据少就永远排不上

### Q6: 如果文章量增长到 10 万篇怎么办？

当前设计的扩展路径：
- 内容相似度：只计算同分类内的文章对，避免全量笛卡尔积
- 协同过滤：引入 LSH（局部敏感哈希）近似最近邻，降低计算复杂度
- 缓存策略：从全量预计算改为按需计算 + LRU 淘汰
- 终极方案：引入 Elasticsearch 做向量检索，或迁移到专业推荐引擎

### Q7: 如何评估推荐效果？

- 离线指标：准确率（Precision@K）、召回率（Recall@K）、NDCG
- 在线指标：推荐模块点击率（CTR）、推荐文章的阅读完成率、用户停留时长
- A/B 测试：对比推荐开启/关闭、不同权重配比的效果差异

## 11. 推荐系统 v2：管理员干预能力（推荐度功能）

**日期**: 2026-05-07
**状态**: 已上线
**版本**: v2

### 11.1 背景与目标

v1 版本通过三路引擎（内容 40% + 协同过滤 40% + 热度 20%）自动计算每篇文章的推荐分。但管理员缺少对推荐结果的干预手段：

- 无法主推一篇冷门但优质的文章
- 无法把某篇文章从推荐位拿下
- 无法标注"精华"让用户在推荐区感知到

v2 引入**推荐度（recommend_level）**字段，让 admin 在发布/编辑文章时指定 0-5 的推荐等级，作为全局乘数影响推荐排序。

### 11.2 推荐度语义

| 等级 | 语义 | 乘数 | 前端标识 |
|------|------|------|----------|
| 0 | 不推荐（屏蔽） | 从候选池排除 | 无 |
| 1 | 弱 | ×0.70 | 无 |
| 2 | 较弱 | ×0.85 | 无 |
| 3 | 默认 | ×1.00 | 无 |
| 4 | 推荐 | ×1.15 | 灰色徽章 |
| 5 | 精华 | ×1.30 | 主色徽章 |

**乘数公式**：
```
multiplier(level) = 1.0 + (level - 3) × 0.15
```

**不影响的场景**：
- 文章详情页、分类页、标签页、搜索结果、文章列表等「非推荐」位置

### 11.3 关键设计决策

| 决策点 | 选择 |
|-------|------|
| 推荐度取值 | 0-5 整数，默认 3 |
| 生效位置 | 作为全局乘数，应用于所有推荐路径（匿名 / 冷启动 / 活跃用户） |
| 推荐度 0 的语义 | 硬屏蔽，从候选池直接排除 |
| 缓存更新策略 | 修改后立即清除所有推荐缓存 |
| 前端视觉展示 | 仅在推荐卡片上，4 档"推荐"徽章、5 档"精华"徽章 |
| admin 入口 | 发布/编辑表单 + admin 文章列表页快捷编辑 |

### 11.4 推荐算法改动

在三个推荐计算方法中应用乘数和屏蔽逻辑：

```java
// 乘数计算
private double recommendMultiplier(Article article) {
    Integer level = article.getRecommendLevel();
    if (level == null) return 1.0;
    return 1.0 + (level - RECOMMEND_LEVEL_DEFAULT) * RECOMMEND_LEVEL_STEP;
}

// 屏蔽判断
private boolean isBlocked(Article article) {
    Integer level = article.getRecommendLevel();
    return level != null && level == RECOMMEND_LEVEL_BLOCKED;
}

// 应用示例
for (Article article : allArticles) {
    if (isBlocked(article)) continue;  // 硬屏蔽
    // ... 计算各路得分 ...
    finalScores.put(article.getId(), baseScore * recommendMultiplier(article));
}
```

### 11.9 面试高频问题（推荐度扩展）

**Q8: 如何平衡算法自动化和人工干预？**

推荐度字段是一种"管理员权重"：
- 算法负责 95% 的场景，保证推荐质量下限
- 人工干预负责 5% 的特殊场景（如推广活动内容、屏蔽敏感文章）
- 两者结合：<u>最终分数 = 算法分 × 推荐度乘数</u>

**Q9: 为什么选择 0-5 而不是更细的粒度？**

- 粒度太粗（0/1）：无法表达"程度"差异
- 粒度太细（0-10）：增加决策成本，收益递减
- 0-5 是产品化友好和表达力度的平衡点

**Q10: 如何避免管理员滥用推荐度？**

- 仅影响推荐位，不影响文章详情页等自然流量
- <u>乘数范围有限（0.7-1.3），无法完全压制优质内容</u>
- 日志记录修改行为，Git 历史可追溯
