## Hazenix Blog 项目优化指南

本文档结合当前代码结构与技术栈（Vue3 + Element Plus + Tailwind / 自定义样式 + Markdown 渲染），给出一份整体优化建议，供后续迭代时参考。

---

## 一、整体工程与架构

- **目录与模块划分**
  - 将通用业务模块（文章、分类、标签、评论、用户等）在前端按 `views` / `components` / `api` / `utils` / `hooks` 等维度拆分清晰，降低单文件体积，避免「巨组件」。
  - 为路由、API、状态管理（如 Pinia）分别建立独立目录，并在根目录加上简要的 `README` 说明模块职责。

- **依赖与版本管理**
  - 固定核心依赖版本（Vue、Vue Router、Element Plus、Markdown 相关库等），避免 minor/patch 自动升级引发线上不兼容。
  - 定期 `npm outdated` / `pnpm outdated`，评估升级收益与风险，记录在 `CHANGELOG` 或文档中。

- **类型与规范**
  - 推荐引入 TypeScript 或在现有 JS 基础上逐步迁移高价值模块（例如 `api`、`utils`、核心 `components`）。
  - 在接口层统一声明类型（Article、Category、Tag 等），避免在组件里到处 `any` 或隐式字段。

---

## 二、前端性能优化

- **加载与渲染性能**
  - 对 `ArticleList` 这类列表页：
    - 如果文章数很多，考虑引入虚拟列表（virtual scroll）或分页接口（当前为前端分页，可在后台分页后再做前端增强）。
    - 对列表卡片中的封面图添加懒加载（如 `loading="lazy"` 或使用指令/组件封装）。
  - 使用路由懒加载，将每个页面打包成独立 chunk，减少首屏 JS 体积。

- **图片与静态资源**
  - 将头像、封面等常驻图片做体积压缩，并使用合适的格式（PNG/JPEG/WebP）。
  - 对公共图标优先使用 Icon Font / SVG Sprite，而不是多张图片。

- **Markdown 渲染优化**
  - 将 Markdown 渲染库（如 `markdown-it`、高亮库等）打到独立 chunk 或通过动态 import 延迟加载，只在文章详情页使用。
  - 对复杂 DOM 操作（目录、锚点跳转、高亮处理等）集中在单一组件内完成，确保外层组件渲染简单，避免不必要的重复计算。

- **缓存与请求优化**
  - 对文章列表、分类、标签等相对静态的数据加缓存策略：
    - 前端：短时间内重复访问时直接使用内存缓存或 Pinia store 中的数据。
    - 后端：对列表接口添加合理的缓存与分页，避免一次性返回过大数据。

---

## 三、代码结构与可维护性

- **组件拆分与复用**
  - 将通用 UI 抽离为独立组件：
    - 文章列表卡片（article-card）
    - 标签展示（tag-list / tag-chip）
    - 通用的 Loading / Empty 状态组件
  - 对 Markdown 渲染中的目录、代码块复制按钮、图片查看等可以拆为子组件或 composable。

- **逻辑抽离**
  - 将与业务无关的逻辑抽到 `utils` 或 `composables`：
    - 日期、数字、摘要等格式化函数（当前已在 `helpers` 中，可进一步补充统一）。
    - SEO 设置逻辑抽取为 `useSEO` / `seo.ts` 工具，复用在各页面。
    - 路由参数解析与筛选逻辑抽离为 `useArticleFilter` 等 composable，避免在单个视图文件中堆积过多逻辑。

- **错误处理与兜底**
  - 为所有 API 调用增加统一错误处理层，在 `api` 封装中处理网络错误、业务错误，并统一弹出提示或返回兜底数据。
  - 对 Mock 数据仅在开发环境下启用，并确保不会无意带到生产逻辑中。

## 四、用户体验（UX）与交互

- **列表与详情体验**
  - 列表页卡片点击区域清晰，支持键盘可访问性（如 `tabindex` 与 `enter` 触发）。
  - 文章详情页支持目录悬浮固定（sticky TOC），阅读进度条等增强阅读体验的功能。

- **深色模式与适配**
  - 当前已经有 `dark` 相关样式类，建议统一通过全局状态或 prefers-color-scheme 控制深色模式，保证各页面风格一致。
  - 对移动端做更细致的适配：标题字号、间距、列表密度、分页组件在小屏下的展示。

- **可用性细节**
  - 为外部链接统一添加 `target="_blank"` 并加上清晰的外链提示（当前 Markdown 渲染中已有样式，可在渲染逻辑上保证一致）。
  - 对复制按钮、目录跳转等操作增加轻量提示，保证用户有明确反馈。

---

## 五、SEO 与可观测性

- **SEO 优化**
  - 为主要页面统一设置：
    - `title`、`meta description`、`keywords`
    - Open Graph / Twitter meta（分享预览）
  - 对文章详情：
    - 使用文章标题作为 `<title>`，首段内容或摘要作为 `description`。
    - 为文章封面配置 `og:image` 等，提升分享展示效果。
  - 搜索引擎友好：
    - 若后端支持，考虑配合 SSR / 预渲染（如 Vite SSR 或静态导出）提高爬虫可见性。

- **埋点与监控**
  - 关键行为埋点：页面浏览、文章阅读、搜索与筛选行为。
  - 集成前端监控（如 Sentry）捕获线上报错，结合后端日志，形成完整链路。

## 六、安全与稳定性

- **XSS 与内容安全**
  - Markdown 渲染严格控制允许的 HTML 标签和属性，对用户可控内容进行 XSS 过滤。
  - 配置合理的 CSP（Content Security Policy），限制脚本与资源来源。

- **接口与权限**
  - 区分前台公开接口与管理后台接口，必要时在前端对敏感操作做简单防重与幂等（例如避免重复提交）。
  - 使用 HTTPS 部署，避免明文传输。

- **错误提示与降级**
  - 当文章列表、详情等关键接口失败时，提供友好提示与重试按钮，而不仅是 Toast。
  - 对 Markdown 渲染失败、代码高亮异常等情况给出降级展示，而不是整块空白。

---

## 七、构建与部署优化

- **打包与构建**
  - 使用 Vite / Webpack 的代码拆分策略（dynamic import）减小首屏包体积。
  - 开启生产环境压缩（gzip / brotli），并在服务器上正确配置 `Content-Encoding`。

- **静态资源缓存**
  - 为静态资源启用强缓存 + 文件名 hash，确保上线后更新可控。
  - HTML 与接口响应使用较短的缓存或协商缓存（ETag / Last-Modified），避免旧页面难以刷新。

---

## 八、文档与协作

- **项目文档**
  - 在 `documents` 目录下维持：
    - 功能设计、数据库设计、安全审计、错误码定义（已存在）。
    - 「本优化指南」与后续的「开发规范」/「接口约定」。
  - 对重要技术决策（如 Markdown 渲染方案、目录生成规则、SEO 策略等）做简短 ADR 记录，方便后续迭代。

- **开发流程**
  - 建议采用 Git 分支策略（如 feature/xxx、fix/xxx），配合代码审查提高代码质量。
  - 为关键模块补充单元测试或 e2e 测试，保证重构时行为稳定。

---

以上建议不要求一次性全部落实，可以结合当前迭代计划，优先处理：**列表与详情体验优化、Markdown 渲染与性能、SEO 与基础监控** 三个方面，再逐步推进工程化和安全层面优化。
